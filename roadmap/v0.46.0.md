# v0.46.0 — Operational Readiness, Scalability & CI Completeness

**Status: Planned** | **Scope: Large**

---

## What is this?

The assessment found that operational deployment assets are demo-grade rather
than production-ready: Dockerfiles use stale VERSION defaults, containers lack
HEALTHCHECKs, CNPG examples underprovision workers, worker exhaustion is only
visible through logs, shared-memory capacities are hardcoded, WAL slot capacity
is reactive, cross-platform CI is non-blocking, and full E2E coverage is not
continuously measured. v0.46.0 closes all of these gaps.

---

## Container and deployment hardening

Dockerfile VERSION args are synced to Cargo.toml at build time. All runtime
images get `pg_isready` HEALTHCHECKs. CNPG ships separate dev and production
examples with explicit worker budgets, resource requests, WAL sizing, and
monitoring hooks.

---

## Preflight and runtime status functions

`pgtrickle.preflight()` checks worker slots, replication slots, WAL level,
shared_preload_libraries, and scheduler state. `pgtrickle.worker_pool_status()`
exposes pool size, active workers, queued jobs, and blocked reasons. Both are
SQL-callable so operators have runtime visibility without log scraping.

---

## Scalability infrastructure

The invalidation ring capacity (currently 128) becomes preload-configurable
with overflow metrics. Worker-slot exhaustion becomes SQL-visible status.
WAL slot capacity is checked proactively before enabling WAL/auto mode. Citus
failure counters persist across scheduler restarts.

---

## CI completeness

Cross-platform compile failures become blocking on scheduled runs. A small
required full-image smoke test runs on PR/push. E2E coverage schedule is
restored. Weekly E2E coverage tracks module-level changes.
